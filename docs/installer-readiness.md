# Installer readiness report

_Date: 2026-06-14 · Scope: `packaging/install.sh` + `packaging/uninstall.sh` + `packaging/soundsync-cod-watch.sh` + `packaging/systemd/*` + `packaging/default/soundsync` + `debian/*` + `src/doctor.rs`. Analysis only; no files modified. The .deb was inspected, not install-tested; the shell path was validated live on Ubuntu 26.04 x86._

This report synthesizes six per-dimension audits (fresh-shell, fresh-deb, re-install/upgrade, uninstall/purge, troubleshoot, dependency completeness). Findings are grounded in `file:line` and spot-verified against the actual source before writing.

---

## Verdict — is it 100% ready?

| Scenario | Status | One-line reason |
|---|---|---|
| **Fresh install — `install.sh`** | **GAPS** | Hard-depends on `hciconfig` that the deps step never installs (FI-1); `set -e` + `enable --now` aborts the whole install on any adapter error (FI-2); never starts the daemon in-run (FI-3); raw-root invocation silently yields a dead appliance (FI-4). Works only on the validated, fully-provisioned 26.04 box. |
| **Fresh install — `.deb`** | **GAPS (critical)** | `postinst` resolves the appliance user to **root** because it sources the conffile and reads ambient `$USER` (DEB-01) → linger enabled for root, **headless daemon dead at boot** on the canonical `sudo apt install` path. |
| **Re-install / upgrade** | **GAPS** | `.deb` prerm disables/stops ALL units on `upgrade` (REUP-01); `install.sh` re-run never restarts the system adapter/CoD units so recovery runs stale code (REUP-02); non-atomic webui clear-before-copy can destroy a working UI mid-recovery (REUP-03). |
| **Uninstall / purge** | **GAPS** | Purge targets a bogus WP Lua path so the real WP 0.4.x config is orphaned (UNINST-1); neither uninstaller stops the running user daemon before deleting its binary (UNINST-2/3). Reaches a working state only on WP 0.5 + after reboot. |
| **Troubleshoot — `--doctor` / `--reset`** | **GAPS** | Hard-aborts unless run as the sudo'd appliance user (DOC-04); validates WP-config *presence* not content/role drift (DOC-02); hardcodes `hci0` (DOC-01); skips port-in-use; assumes the product is already installed (DOC-05). |

### Overall verdict: **NOT 100% ready.**

On the single validated machine (Ubuntu 26.04, WP 0.5, single sudo user with an active session, all deps present, adapter = hci0) every path works. But the installer is **not robust on a fresh, partial, non-default-user, or non-26.04 machine**, and the canonical `.deb` install path has a **critical** functional break (DEB-01) that produces a dead appliance at boot. There are **3 critical** and **8 high** gaps to close before declaring readiness across target machines.

De-duplicated across dimensions. Where multiple audits found the same root cause, they are merged and cross-referenced. Effort: **S** ≈ <1h, **M** ≈ a few hours, **L** ≈ a day+.

## Assumptions ledger

Every assumption the installer paths make, with how it is handled.

| Assumption | Status |
|---|---|
| Invoked via `sudo` by the actual appliance user, so `SUDO_USER` is set and non-root (`install.sh:93`) | **Silent failure** — raw-root yields a dead appliance (G5); no detection or warning |
| `.deb` maintainer scripts run with `USER=root` ambient (the canonical `sudo apt install` env) | **Silent failure** — leaks into SS_USER → linger for root (G1) |
| A Bluetooth adapter is present **and `hciconfig` exists** at install time (`install.sh:498`) | **Silent failure** — `hciconfig` not in deps (G2); error aborts install (G3) |
| Host is Ubuntu 26.04 with identical apt package names; machine is online for `apt-get update` (`install.sh:338,344-356,370`) | **Silent failure** — name drift (N2) and offline `update` (N1) abort the install |
| `pipewire` transitively pulls in `pipewire-bin` (the pw-* tools) | **Silent failure** — not contractual across releases (G11) |
| node/npm system-installed or via `SOUNDSYNC_NODE_BIN`; `~/.local/node/bin` resolves under sudo | **Documented but broken** — README claims it; HOME=/root under sudo defeats it (N3) |
| The target user has, or will after login/linger have, an active `/run/user/<uid>` session | **Documented** — install does not start the daemon in-run; "Next steps" admits relogin (G4) |
| `wireplumber` is installed before `apply-wireplumber-config` runs, so format detects correctly | **Silent failure** — falls back to Lua on probe failure, writes the wrong-format file (FI-8 / part of G10) |
| Port 8080 is the bind; the install-time port check is advisory only | **Handled at install** (`install.sh:516-531`); **silent in doctor** (G17) |
| WP config lives only in `/etc/wireplumber`; a present file + marker line == correct config | **Silent failure** — content/role drift and user-dir shadow configs undetected (G10, N6) |
| WP config paths in `src/wireplumber/config.rs` are the only two the binary writes; purge must match exactly | **Silent failure** — purge targets a bogus path (G6) |
| Single-appliance-user headless box: exactly one human uid≥1000 with a login shell | **Handled** in postinst auto-detect (`:59-69`) **but bypassed** by G1; install.sh has no equivalent (G5) |
| dpkg upgrade ordering = old-prerm(upgrade) → unpack → new-postinst(configure) | **Handled** (correct), but prerm over-acts on `upgrade` (G12) |
| `systemctl enable --now` does not restart an already-active unit (systemd semantics) | **Handled assumption**, but its consequence (stale system units on re-run) is unaddressed (G13) |
| `systemctl --global enable` symlinks are NOT tracked by deb-systemd-helper state | **Silent leak** on purge (N12) |
| Invoked via `sudo install.sh --doctor/--reset` with SUDO_USER = the single appliance user | **Silent failure / hard-abort** for root-shell/su recovery (G8) |
| The product is already installed at the install.sh layout when doctor/reset run | **Silent no-op** on a partial/broken install (G9) |
| Adapter is `hci0` for the daemon's own doctor | **Silent false-Warn** on hci1+ (G7) |
